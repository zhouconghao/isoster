### Model Reconstruction Audit
Reviewed `isoster/model.py` `build_isoster_model()` and identified two issues:

## In Progress (Not Finished)

### Issue 1: `harmonic_orders` default silently drops higher orders
- `build_isoster_model()` defaults `harmonic_orders=[3, 4]` (line 75 of `model.py`)
- If fitting used `harmonic_orders=[3,4,5,6,7]`, calling without explicit orders drops 5-7 silently
- The example `shared.py` (lines 301-308) already works around this by auto-detecting from keys
- **Fix**: Auto-detect available orders from isophote dict keys when `harmonic_orders=None`
- Scan for keys matching pattern `a{n}` where n >= 3 in the first valid isophote

### Issue 2: EA mode angle space mismatch in model reconstruction
- When `use_eccentric_anomaly=True`, harmonics are fitted in ψ-space (eccentric anomaly)
- But `build_isoster_model()` uses `theta = arctan2(y_rot, x_rot)` (line 206) which is φ (position angle)
- Correct ψ would be `arctan2(y_rot / (1 - eps), x_rot)`
- This is a **pre-existing issue** affecting both ISOFIT and post-hoc paths equally
- Practical impact grows with ellipticity; matters most for eps > 0.5
- **Fix**: Either:
  - (a) Add `use_eccentric_anomaly` parameter to `build_isoster_model()` and switch angle computation, OR
  - (b) Store the angle mode in the isophote results dict during fitting and auto-detect it

## Problems / Blockers
None. Both issues are well-understood with clear fix paths.

## Key Decisions
1. **Auto-detect pattern**: Scan isophote keys for `a{n}` pattern rather than requiring explicit `harmonic_orders` parameter
2. **EA angle fix approach**: Needs user input on whether to add a parameter or auto-detect from results
