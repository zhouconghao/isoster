## Part 1: Design Issue

### R26-05. Forced photometry API needs unification with template mode

**Files:** `isoster/config.py:102-104`, `isoster/driver.py:92-163, 276-344`

The current API has two overlapping mechanisms for forced photometry:

1. **`forced=True` + `forced_sma=[...]`** (config-based): Uses a single fixed geometry (x0, y0, eps, pa from config) at specified SMA values. The geometry is the same for all SMA values.

2. **`template_isophotes=[...]`** (function parameter): Uses variable geometry per SMA from a previous isoster run. Each SMA gets its own (x0, y0, eps, pa) from the template.

**Problems with the current design:**

- The `forced` mode with a single fixed geometry is nearly useless in practice — real galaxies have varying geometry with radius. The useful workflow is always template-based.
- The V6 warning (`template_isophotes takes priority over forced=True`) acknowledges the overlap but doesn't resolve it.
- The naming is confusing: both mechanisms do "forced photometry" but through different code paths with different capabilities.
- Template isophotes can only be passed as an in-memory list. Users must manually call `isophote_results_from_fits()` to load a template file. The common multiband workflow should accept a file path directly.
- The `forced_sma` config field is a vestigial concept — SMA values should come from the template, not be specified separately.

**Proposed redesign (for discussion):**

Replace `forced=True`/`forced_sma` with a single `template` parameter that accepts:
- A file path (string) → loads via `isophote_results_from_fits()`
- A results dict (from a previous `fit_image()` call) → extracts `['isophotes']`
- A list of isophote dicts (current `template_isophotes` behavior)

The config fields `forced` and `forced_sma` would be deprecated. The `template_isophotes` function parameter would be renamed to `template` for brevity.

```python
# New API (proposed)
results_r = isoster.fit_image(image_r, mask_r, config,
                               template="galaxy_g.fits")
# or
results_r = isoster.fit_image(image_r, mask_r, config,
                               template=results_g)
# or (current behavior, still supported)
results_r = isoster.fit_image(image_r, mask_r, config,
                               template=results_g['isophotes'])
```

**Scope:** This is an API design change that should be discussed before implementation. It affects `config.py`, `driver.py`, `__init__.py` exports, `CLAUDE.md`, user-facing docs, and possibly `cli.py`.

## Part 2: Important Issues

### R26-01. Second-gradient formula still wrong for linear growth mode

**File:** `isoster/fitting.py:642-645`

The first gradient was correctly fixed in I3 (line 606-607):
```python
delta_r = step if linear_growth else sma * step
gradient = (mean_g - mean_c) / delta_r
```

But the second-gradient fallback path (lines 642-645) was NOT fixed:
```python
gradient = (mean_g2 - mean_c) / sma / (2 * step)          # line 642
gradient_error = (...) / sma / (2 * step)                  # lines 644-645
```

For linear growth, `gradient_sma_2 = sma + 2 * step`, so `delta_r = 2 * step`. The formula should be `/ (2 * step)`, not `/ sma / (2 * step)`. Current code is wrong by a factor of `sma`.

For geometric growth, `delta_r ≈ sma * 2 * step`, so `/ sma / (2 * step)` is correct.

**Fix:**
```python
delta_r_2 = 2 * step if linear_growth else sma * 2 * step
gradient = (mean_g2 - mean_c) / delta_r_2
gradient_error = (...) / delta_r_2
```

**Impact:** Only affects linear growth mode (`linear_growth=True`) when the second-gradient path is triggered. Geometric growth (the default) is unaffected. Low practical impact since linear growth is rarely used, but it's a correctness issue.

### R26-02. `extract_forced_photometry` hardcodes harmonic keys regardless of config

**File:** `isoster/fitting.py:100-103, 119-127`

`extract_forced_photometry` always includes `a3, b3, a3_err, b3_err, a4, b4, a4_err, b4_err` in its return dict — both in the empty-data early return (line 100-103) and the success return (line 119-127). These are always zero since forced photometry doesn't fit harmonics.

This is inconsistent with the V8 cleanup where `fit_isophote` now only includes harmonic keys when `compute_deviations` or `simultaneous_harmonics` is enabled. The V8 test (`test_no_harmonic_keys_when_disabled`) validates that `fit_isophote` doesn't produce these keys, but forced photometry still does.

**Impact:** Not a correctness bug — the zeros are benign. But it breaks the V8 contract that harmonic keys are absent when deviations are disabled. This inconsistency could confuse downstream code that uses key presence to detect whether harmonics were fitted.

**Note:** This issue is subsumed by R26-05 — a forced photometry API redesign would address the output schema as part of the unification.

### R26-03. Post-hoc harmonic deviation code duplicated three times

**File:** `isoster/fitting.py:998-1025, 1098-1119, 1142-1164`

The identical 20-line block for computing post-hoc harmonic deviations appears at:
1. **Harmonic convergence** (line 998-1025)
2. **Geometry convergence** (line 1098-1119)
3. **Stop code 2 fallback** (line 1142-1164)

All three blocks contain:
- The `use_best_data` flag logic
- The `posthoc_angles/posthoc_intens/posthoc_gradient` selection
- The simultaneous vs sequential branching
- The per-order result storage loop

This violates DRY and is a maintenance hazard — any future change to the harmonic computation logic must be applied to all three locations, or they will diverge.

**Fix:** Extract a helper function:
```python
def _compute_posthoc_harmonics(best_geometry, angles, intens, gradient,
                                best_angles, best_intens, best_gradient,
                                sma, harmonic_orders, isofit_mode,
                                simultaneous_harmonics):
    ...
```

### R26-04. V5 warning fires for every forced-mode config due to defaults

**File:** `isoster/config.py:249-266`

The V5 validation checks whether `compute_errors=True` and `compute_deviations=True` when `forced=True`, and warns that these are "ignored". But both are **defaults** — `compute_errors` defaults to `True` and `compute_deviations` defaults to `True`.

This means `IsosterConfig(forced=True, forced_sma=[5.0, 10.0])` ALWAYS produces a warning:
```
UserWarning: forced=True ignores: compute_errors, compute_deviations
```

The test `test_no_warn_forced_defaults` (line 78-81) confirms this is intentional by *expecting* the warning. However, it's noisy for users.

**Note:** This issue is subsumed by R26-05 — a forced photometry API redesign would eliminate the `forced` config flag entirely.

## Part 8: plot_qa_summary Direct Residual Change

The change from fractional to direct residual as default is well-implemented:
- `relative_residual=False` (new default): computes `image - isoster_model`
- `relative_residual=True` (old behavior): computes `100 * (model - data) / data`
- Color scale limits derived from data (99th percentile for direct, clipped [0.05, 8.0] for fractional)
- Colorbar label correctly updates between modes

Note: the sign convention differs between modes — direct residual uses `data - model` while fractional uses `model - data`. This should be documented.
