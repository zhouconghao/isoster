    Isoster Milestone Code Review -- 2026-02-22

    Comprehensive review of the isoster codebase at milestone: Huang2013 benchmark campaign completed, basic development reaching maturity. Review covers core library, test suite, examples, documentation, and packaging.

    Executive Summary

    The isoster codebase is well-architected with clean module separation, correct mathematical foundations (Jedrzejewski 1987 formalism, Ciambur 2015 EA mode), and a solid test harness. The refactoring from monolithic optimize.py into sampling.py, fitting.py, driver.py, and model.py is well done. Pydantic config, numba fallback patterns, and post-run validation are all good engineering decisions.

    That said, the review uncovered 1 critical bug (EA + sigma clipping), several important correctness issues, significant test coverage gaps, and documentation staleness that should be addressed before the next development phase.

    Severity Count Summary
    Critical 1 EA mode sigma clipping breaks phi/intens correspondence
    Important 16 Correctness, robustness, test quality, docs staleness
    Minor/Suggestion 20+ Code quality, test organization, docs completeness

    Part 1: Core Library Review

    CRITICAL

    C1. EA mode + sigma clipping causes angle/intensity mismatch -- FIXED

    Fixed in: commit 194ad4c (branch fix/ea-sigma-clip-phi-mismatch, merged to main) Regression test: tests/unit/test_fitting.py::test_sigma_clip_ea_phi_sync

    File: isoster/fitting.py:666-667

    In fit_isophote(), sigma clipping is applied to angles and intens:

    angles, intens, n_clipped = sigma_clip(angles, intens, sclip, nclip, sclip_low, sclip_high)

    But phi (used for geometry updates and error computation) is not clipped in parallel. After clipping, phi retains its original length while angles and intens are shorter.

    • In regular mode (non-EA), angles == phi, so both are effectively clipped together.
    • In EA mode, angles is psi (eccentric anomaly) and phi is a different array. After clipping, phi has N elements while intens has M < N elements.

    This causes mismatched arrays passed to compute_gradient(current_data=(phi, intens)) (line 712) and compute_parameter_errors(phi, intens, ...) (line 756).

    Impact: In EA mode with sigma clipping enabled (nclip > 0), gradient and error computations receive mismatched arrays, leading to silent numerical errors or incorrect results.

    Fix applied: sigma_clip now accepts extra_arrays parameter; EA mode passes phi through it so the same mask is applied to all arrays.

    IMPORTANT

    I1. fflag check uses inverted semantics from photutils convention -- FIXED

    Fixed in: branch fix/fflag-and-var-residual-guard Regression tests: tests/unit/test_fitting.py::TestFflagSemantics (3 tests)

    File: isoster/fitting.py:669

    if actual_points < (total_points * fflag):

    The fflag parameter is documented as "Maximum fraction of flagged data." With photutils convention, fflag=0.2 means "reject if >20% flagged", so the check should be actual_points < total_points * (1.0 - fflag).

    Current behavior: fflag=0.2 allows up to 80% flagging (inverted from user expectation). A strict fflag=0.2 setting is actually very permissive.

    Fix applied: Changed to actual_points < total_points * (1.0 - fflag). Note: isoster's fflag check only applies to sigma-clipped points (masking is handled earlier in sampling), unlike photutils where total_points includes mask-rejected points.

    I2. var_residual can be NaN/Inf when data points <= parameters -- FIXED

    Fixed in: branch fix/fflag-and-var-residual-guard Regression tests: tests/unit/test_fitting.py::TestVarResidualDdofGuard (5 tests)

    File: isoster/fitting.py:236, 296, 381

    var_residual = np.std(intens - model, ddof=len(coeffs))**2

    When len(intens) <= len(coeffs), np.std(..., ddof=N) has a zero or negative denominator, producing nan or inf. This propagates silently through error computations.

    Same issue in compute_deviations (line 296) and fit_higher_harmonics_simultaneous (line 381).

    Fix applied: Added early-return guards (len(intens) <= n_params → return zeros) in compute_parameter_errors (top-level + both code paths), compute_deviations, and fit_higher_harmonics_simultaneous.

    I3. Gradient formula incorrect in linear growth mode -- FIXED

    Fixed in: branch fix/gradient-linear-and-pa-unwrap Regression tests: tests/unit/test_fitting.py::TestGradientLinearGrowth (2 tests)

    File: isoster/fitting.py:480

    gradient = (mean_g - mean_c) / sma / step

    For geometric growth: delta_r = sma * step, so gradient = (mean_g - mean_c) / (sma * step) -- correct.

    For linear growth: delta_r = step, so gradient should be (mean_g - mean_c) / step, but the code divides by sma * step -- incorrect by a factor of sma.

    Fix: 

    delta_r = sma * step if not linear_growth else step
gradient = (mean_g - mean_c) / delta_r

    I4. model.py does not filter NaN intensities before interpolation -- FIXED

    Fixed in: branch fix/gradient-linear-and-pa-unwrap Regression tests: tests/unit/test_model.py::TestModel::test_nan_isophotes_filtered_before_interpolation, test_nan_geometry_filtered_before_interpolation, test_nan_harmonic_coefficients_treated_as_zero

    File: isoster/model.py:103-104

    Isophotes with stop_code != 0 may have intens = np.nan. These are included in the interpolation arrays, causing interp1d to produce NaN across the model.

    Fix applied: Filter isophotes requiring finite values in all 5 key columns (intens, x0, y0, eps, pa). Additionally, non-finite harmonic coefficients are replaced with 0.0 to prevent poisoning the harmonic interpolation.

    I5. PA interpolation in model.py wraps incorrectly at 0/180 boundary -- FIXED

    Fixed in: branch fix/gradient-linear-and-pa-unwrap Regression test: tests/unit/test_model.py::TestModel::test_pa_interpolation_wrap_boundary

    File: isoster/model.py:140

    Linear interpolation of position angle values that wrap (e.g., 170 to 10 degrees) produces incorrect intermediate values (~90 degrees) instead of interpolating through the boundary.

    Fix: Apply np.unwrap() to PA values before interpolation.

    I6. add_cog_to_isophotes has no length check against CoG results -- FIXED

    Fixed in: branch fix/gradient-linear-and-pa-unwrap Regression test: tests/integration/test_edge_cases.py::TestAddCogLengthCheck::test_length_mismatch_raises

    File: isoster/cog.py:188-193

    If the input list lengths don't match, cog_results['cog'][i] raises an IndexError.

    Fix applied: Added explicit length check raising ValueError with a clear message before the loop.

    I7. Missing valid field in forced photometry results -- FIXED

    Fixed in: branch fix/gradient-linear-and-pa-unwrap Regression test: tests/integration/test_edge_cases.py::TestForcedMode::test_forced_photometry_has_valid_field_on_failure

    File: isoster/fitting.py:117-126

    extract_forced_photometry returns dicts without a valid field, while fit_central_pixel includes it. Consumers checking for this field will get a KeyError.

    Fix applied: Added 'valid': True to the success return and 'valid': False to the stop_code=3 early return.

    I8. Stop code 2 path does not compute harmonic deviations -- FIXED

    Fixed in: branch fix/gradient-linear-and-pa-unwrap Regression test: tests/unit/test_fitting.py::test_stop_code_2_computes_harmonic_deviations

    File: isoster/fitting.py:842-845

    When maxit is reached (stop_code=2), the code attaches photometry but NOT higher-order harmonic deviations. a3, b3, a4, b4 are all 0.0 for these isophotes, even though a best-effort computation is available from the last iteration.

    Fix applied: Added best-effort harmonic computation (both sequential and simultaneous paths) in the stop_code=2 block, using the last iteration's angles, intens, and gradient.

    Critical Test Issues

    T1. test_model_residuals.py has no assertions -- always passes

    File: tests/validation/test_model_residuals.py:270-276

    The pass/fail logic only prints results, never asserts. pytest reports PASS regardless of residual quality.

    # Prints but never asserts:
if mid_max < 1.0 and mid_median < 0.5:
    print(f"EXCELLENT: ...")
else:
    print(f"POOR: ...")  # Still passes!

    T2. Non-deterministic random seeds in test_fitting.py

    File: tests/unit/test_fitting.py:127, 183, 474, 201

    Multiple test functions use np.random.random() without setting a seed, making tests non-reproducible. Some tests in the same file correctly use np.random.seed(42).

    T3. Dual __main__ blocks in test_fitting.py

    File: tests/unit/test_fitting.py:95-96, 297-298

    The file has both unittest.main() (line 95) and pytest.main() (line 297). The unittest one executes first, never reaching the pytest block.

    T4. create_sersic_model duplicated in 4+ test files

    The function is defined in tests/fixtures/sersic_factory.py (canonical) but copied with slight variations into test_sersic_accuracy.py, test_numba_validation.py, test_model_residuals.py, and test_photutils_comparison.py. Each copy has different implementations (e.g., loop-based vs reshape-mean downsampling).

    Part 3: Documentation Staleness

    IMPORTANT Issues

    D1. CLAUDE.md missing stop code 2

    File: CLAUDE.md:126

    States: "Stop codes: 0=converged, 1=too many flagged pixels, 3=too few points, -1=gradient error"

    Missing: stop code 2 (max-iteration fallback), which was reintroduced in Phase 18.

    D2. README.md acronym conflict

    File: README.md:1

    README says "ISOphote fitting with Speedy Templated Extraction and Regression" while CLAUDE.md says "ISOphote on STERoid". Two different expansions for the same acronym.

    D3. __init__.py does not export IsosterConfig

    The core config type is not available from the top-level isoster namespace. Users must use from isoster.config import IsosterConfig.

    D4. pyproject.toml claims requires-python >= 3.8 -- stale

    File: pyproject.toml:10

    Pydantic v2 and dev dependencies practically require >= 3.9. The requires-python should be updated.

    D5. docs/spec.md missing modules

    The core module list omits output_paths.py, numba_kernels.py, and does not document the ACCEPTABLE_STOP_CODES constant.

    D6. plotting.py diverges from established QA figure rules -- FIXED

    Fixed in: commit 4ac2c44 (branch docs/polish-and-exports, merged to main) Verification: 152 tests pass; M51 QA figure generated at outputs/qa_m51_test.png

    Rewrote plot_qa_summary to follow build_method_qa_figure baseline: stop-code-separated scatter, arcsinh display with colorbars, fractional-percent residuals (coolwarm), PA unwrap normalization, and adaptive panel count. Promoted 15+ reusable helpers from the Huang2013 demo into isoster/plotting.py as canonical location.

    1. Clean module separation -- each module has a clear, focused responsibility
    2. Numba fallback pattern -- dual-implementation with automatic selection is maintainable
    3. EA mode correctness -- psi/phi separation per Ciambur (2015) is correctly implemented
    4. Pydantic config -- excellent for validation, serialization, and documentation
    5. Post-run validation -- _validate_non_negative_error_fields in driver.py is good defensive programming
    6. Geometry updates -- Jedrzejewski (1987) equations correctly implemented with PA wrap and eps bounds
    7. Huang2013 campaign -- well-architected benchmark workflow with fault tolerance and retry logic
    8. Journal/lessons tracking -- good practice maintained throughout development
